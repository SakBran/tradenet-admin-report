# Excel Export Job Queue + Shared Drive

Replaces the synchronous, in-memory Excel export (which OOMs / times out on
large data) with an **asynchronous job queue**: clicking *Excel* enqueues a job
in **TemplateDB**, an in-process background worker generates the file with
**chunked paging** (flat memory) onto a **persistent disk folder**, and the user
downloads it from a **shared "Exports" drive** page. Identical requests reuse a
finished file instead of regenerating.

> Status: **infrastructure + pilot reports done**; remaining 155 controllers
> roll out per the playbook below.

---

## Why the old path fails on large data

`[HttpPost("Excel")]` → `ExcelGenerator.CreateWorkbookAsync` does three things
that don't scale (`Backend/Service/Reports/ExcelGenerator.cs`):
1. `ApiResult.CreateAsync` → `ToListAsync()` materializes the **entire** result
   set in memory (+ a full `CountAsync()` scan first).
2. The workbook is built in a `MemoryStream` then `.ToArray()` — a **second**
   full copy as `byte[]`.
3. It's a **synchronous** HTTP request, so long generations hit proxy / Kestrel
   / browser timeouts even when memory holds.

The new path fixes all three: stream rows in chunks, write straight to a
`FileStream` on disk, and run it in the background.

## Architecture

- **Queue store:** table `ExcelExportJobs` in **TemplateDB** (via
  `ApplicationDbContext` + EF migration). Holds status, filter hash, request
  JSON, file path, sizes, timestamps, lease.
- **Worker:** `ExcelExportWorker : BackgroundService` polls for `Queued` jobs,
  claims one with a **lease** (so only one worker instance runs it),
  generates, updates status. Concurrency-limited. On startup, jobs left
  `Processing` past their lease are requeued.
- **Cleanup:** `ExcelExportCleanupWorker : BackgroundService` deletes files +
  rows past `ExpiresAtUtc`.
- **Generation:** `StreamingExcelWriter` pulls rows in **chunks** (OFFSET/FETCH),
  resolves reference lookups (country CSV → names) per chunk in C# — matching the
  existing `_Fast` pattern — appends each chunk to the worksheet `XmlWriter`
  writing to a `FileStream`, and **rolls over to a new sheet** at 1,048,575 rows.
- **Registry:** a job can't serialize an `IQueryable`, so each report registers a
  handler keyed by `reportKey` that rebuilds its query from the stored request
  JSON. See "Handler kinds".
- **Storage:** files live under `ExcelExport:StorageRoot` (config), **outside
  `wwwroot`** — downloads go through an `[Authorize]` endpoint, never a public
  static URL.

### Job lifecycle

`Queued → Processing → Completed | Failed`; expired rows are deleted by cleanup.
Statuses: `Queued=0, Processing=1, Completed=2, Failed=3`.

## Reuse / dedup rule

On enqueue for `(reportKey, request)`:
1. `filterHash = sha256(reportKey + normalized-request-json)` — normalization
   **drops grid-only fields** (`PageIndex`, `PageSize`, `SortColumn`,
   `SortOrder`, `FilterColumn`, `FilterQuery`, `IncludeTotalCount`) and lower-cases
   keys; only the real report filters + dates contribute.
2. `isPeriodClosed = request.ToDate.Date < today`.
3. Decision:
   - **A `Queued`/`Processing` job with the same hash exists** → return
     `{status:"processing"}` ("already generating, please wait"); do **not**
     enqueue a duplicate.
   - **A `Completed` job with the same hash exists, file on disk, not expired:**
     - `isPeriodClosed` (historical / "use the old one") → **reuse**, return
       `{status:"ready", jobId, downloadUrl}`.
     - not closed ("up-to-date" range touching today, data still changing) →
       **regenerate** (enqueue new) so the user gets the latest data.
   - **Otherwise** → enqueue new, return `{status:"queued", jobId}`.
4. **Retention:** `ExpiresAtUtc = CreatedAtUtc + 1 day`; cleanup deletes after.

> Worked example: historical range 1.1.2026–1.1.2027 requested twice the same day
> → instant reuse. A report ending *today* → always freshly generated.

## Visibility

The Exports drive is **shared**: any authenticated user sees and can download
every job. `RequestedByUserName` (from `ClaimTypes.Name`) is stored for display
only ("Generated by").

## API

Base: `api/ExcelExport` (all `[Authorize]`).

| Verb | Route | Purpose |
| --- | --- | --- |
| POST | `/{reportKey}` | Enqueue (body = the report's request DTO). Returns `{status, jobId, fileName, downloadUrl}`. |
| GET | `/jobs` | List all jobs (shared), newest first. |
| GET | `/{id}` | One job's status (for polling). |
| GET | `/{id}/download` | Stream the file. Verifies it exists on disk first; 410 if missing. |
| DELETE | `/{id}` | Delete file + row. |

## Handler kinds (registry)

Each report registers one `IExcelReportJobHandler` (in
`Backend/Service/ExcelExport/Handlers/`). Three shapes cover all reports:

1. **Simple `IQueryable`** (most reports) — use
   `QueryableExcelReportJobHandler<TRequest, TResult>` with
   `(TradeNetDbContext db, TRequest req) => IQueryable<TResult>`. The handler
   pages it in chunks automatically (applies a deterministic `OrderBy` if the
   query is unordered).
2. **`_Fast` with per-chunk lookup resolution** — a small custom handler that
   pages the raw row queryable, calls the report's `ToResult(countryNames)` per
   chunk, and appends. Mirrors `*_Fast.CreateExcelWorkbookAsync` but chunked.
3. **Aggregate** — custom handler that runs the SQL GROUP BY and appends the
   (already small) grouped result.

`reportKey` convention = controller route name without `Controller`
(e.g. `MemberRegistrationReport`).

## Per-controller rollout (the remaining 155)

For each existing `[HttpPost("Excel")]`:
1. **Register a handler** for its `reportKey` (kind 1/2/3 above).
2. **Replace the action body** with a call to `IExcelExportJobService.EnqueueAsync(reportKey, request, User)` and return its result DTO. (The old direct-`File()` body is removed.)
3. No frontend change needed per-report — `GenericReportPage` handles the
   enqueue/poll/download flow generically off `config.excelRoute`.

Pilot (done): `MemberRegistrationReport` (simple) and
`ImportLicenceDetailReport` (`_Fast` + CSV lookup).
