🚀 TradeNet Admin Report — Deployment Guide

How to point the website to the correct API, allow it in CORS, and publish — for UAT and Production.

💡You only need to get TWO things right
  1. Frontend → API: the website must call the correct API address. (file: Frontend/src/config.ts)
  2. API → Frontend: the API must allow the website's address in CORS. (file: Backend/Program.cs)
These two must match each other. Everything else is just "build and copy the files".

Do not mix them up — this is the #1 cause of "it doesn't work".

🔎 Step 1 — Find the real URLs from IIS bindings

Log in to the server and open IIS Manager (press ⊞ Win, type inetmgr, Enter). The address of each site is in its Bindings.

A

How to read a site's address

  1. In IIS Manager, expand Sites on the left.
  2. Click the site (for example the website, or the API).
  3. On the right side, click Bindings…
  4. Look at the Host Name, Type (http/https) and Port.
🧰 Site Bindings — reportuat (UAT website)
TypeHost NamePortIP Address
httpsreportuat.myanmartradenet.com443*

Read that row like this:

https + reportuat.myanmartradenet.com + port 443  →  https://reportuat.myanmartradenet.com

ℹ️About the port Port 443 = normal https, you don't write it. Port 80 = normal http, you don't write it. Any other port (like 8000) you do write it, e.g. https://myapi:8000.

🖥️ Step 2 — Frontend: set the API URLs in config.ts

File: Frontend/src/config.ts. This is where the website learns which API to call.

What it looks like now

const envConfig = {
  baseUrl:  'https://localhost:8000/api/',   // API for data
  imageUrl: 'https://localhost:8000/Image/',  // API for images
  qrUrl:    'https://uatapi.ecomreg.gov.mm/QR/',
};
export default envConfig;
🚫Problem These point to localhost (your own PC). On the real server the website would try to reach the API on the user's own computer — so no data loads. You must put the real API address here.

✅ Recommended: keep BOTH addresses, auto-pick by website

Replace the whole file with the version below. It holds the Production API and the UAT API, and automatically chooses the right one based on which website the user opened. This means one build works on both servers — no need to edit and rebuild twice.

// ============================================================
//  API ADDRESSES — fill in the REAL urls from IIS (Step 1).
//  Keep the "/api/", "/Image/", "/QR/" endings exactly.
// ============================================================

// ---- PRODUCTION (live) ----
const PRODUCTION = {
  baseUrl:  'https://PROD-API-ADDRESS/api/',
  imageUrl: 'https://PROD-API-ADDRESS/Image/',
  qrUrl:    'https://PROD-API-ADDRESS/QR/',
};

// ---- UAT (testing) ----
const UAT = {
  baseUrl:  'https://UAT-API-ADDRESS/api/',
  imageUrl: 'https://UAT-API-ADDRESS/Image/',
  qrUrl:    'https://uatapi.ecomreg.gov.mm/QR/',   // existing UAT QR — confirm on server
};

// The PRODUCTION website's host name (NO https://, NO slash).
// Find it in IIS bindings for the live website.
const PRODUCTION_WEBSITE = 'PROD-WEBSITE-HOST';  // e.g. 'report.myanmartradenet.com'

const isProduction = window.location.hostname === PRODUCTION_WEBSITE;

const envConfig = isProduction ? PRODUCTION : UAT;
export default envConfig;
✏️Replace every red box with your real addresses from Step 1. Do not change the /api/, /Image/, /QR/ endings, and keep the last slash / — the app glues the rest of the path onto these.
🔁Simpler alternative (if you prefer one address at a time) If you build separately for each environment, you can instead set just one set of values before each build: put the UAT API in baseUrl/imageUrl/qrUrl, build & deploy to UAT, then change them to Production, build & deploy to Production. The "auto-pick" version above avoids this double work, so it is recommended.

📦 Step 3 — Frontend: build the website

In the Frontend folder, run:

cd Frontend
npm install        # only the first time, or after package changes
npm run build      # creates the dist/ folder

This produces the ready-to-publish files in Frontend/dist/. That whole dist folder is what goes onto the website's IIS folder.

✅Check After it finishes you should see Frontend/dist/index.html and a dist/assets/ folder.

🛡️ Step 4 — Backend: add the website URLs to CORS

File: Backend/Program.cs, inside the #region Cors block (around line 142). CORS is the API's "guest list" — only the website addresses on this list are allowed to call the API from a browser.

🚫If a website is NOT on this list the browser blocks it and you see a CORS error in the console (F12), and reports show no data.

Add your two WEBSITE addresses here

builder.WithOrigins(
    "https://vehicle.myanmartradenet.com",
    "https://testingvehicle.myanmartradenet.com",
    "https://www.mpu-ecommerce.com",
    "https://www.mpuecomuat.com",
    "https://reportuat.myanmartradenet.com",    // <- UAT website (already here)
    "https://PROD-WEBSITE-ADDRESS",            // <- ADD the Production website
    "http://localhost:5173",                     // (local dev — leave as is)
    // ... the other localhost lines stay ...
).AllowAnyMethod().AllowAnyHeader().AllowCredentials();
📏Rules for CORS addresses (must be exact)
  • Use the website address (where users open the app) — not the /api/ address.
  • Include https:// (or http://) — must match how the site is opened.
  • No trailing slash and no path: write https://report.myanmartradenet.com, not .../ or .../reports.
  • If UAT website (reportuat...) is already correct, you may only need to add Production.

📦 Step 5 — Backend: build & publish

In the Backend folder, run:

cd Backend
dotnet publish API.csproj -c Release -o publish

This creates a Backend/publish/ folder containing the compiled API ready for IIS.

🔐Don't overwrite the server's settings The live server's appsettings.json (database connection string, JWT key) must not be replaced by your local one. When copying, skip appsettings.json and any appsettings.*.json. (The deploy.ps1 script already excludes these for you.)

🚚 Step 6 — Copy the files to the server

Each environment has its own folder under IIS. You copy the built files into the matching folder, then the site is live.

📂 Frontend
copy everything inside Frontend/dist/ → the website's IIS folder
📂 Backend
copy everything inside Backend/publish/ → the API's IIS folder
⚡Easiest way — use the included script There is already a deploy.ps1 in the project. On the server it: pulls latest code, builds backend & frontend, and copies them to P:\WEBSITES\tradenet-admin-backend and P:\WEBSITES\tradenenet-admin-frontend, while protecting appsettings.json and web.config. Run it from the project root: 
powershell -ExecutionPolicy Bypass -File .\deploy.ps1
Tip: confirm those two target paths match your real IIS site folders before running.
♻️After deploying the API In IIS Manager, recycle the API's Application Pool (or restart the site) so the new CORS settings take effect. For the website, just refresh the browser with Ctrl+F5 (hard refresh).

🩺 If something goes wrong

❌Reports show no data + browser console (F12) shows a "CORS" / "blocked by CORS policy" error
→ The website address is missing or misspelled in CORS (Step 4). Add the exact address, no trailing slash, then recycle the API app pool.
❌Page loads but every request fails / points to localhost
→ config.ts still has the old localhost address, or you forgot to rebuild the frontend (Step 2 & 3).
❌UAT works but Production calls the UAT API (or vice-versa)
→ PRODUCTION_WEBSITE in config.ts does not exactly match the live website's host name from IIS bindings.
❌"Mixed content" error
→ Website is https but the API address is http. Make both https.
❌404 on every API call
→ The baseUrl is missing the /api/ ending or the final slash /. It must end with /api/.
